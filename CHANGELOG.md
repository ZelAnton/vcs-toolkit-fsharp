# Changelog

All notable changes to **VcsToolkit** are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added
- `Repo.NewChild(reference)` on `VcsToolkit.Core` — start new work on top of `reference` without modifying it (git `checkout <reference> --`; jj `new <reference>`, via the new `Jj.NewChild`/`JjAt.NewChild`). On jj this differs from `Checkout`/`jj edit`: it stacks a fresh, undescribed child change on `reference` instead of rewriting `reference`'s own commit in place.
- `Repo.ShowFile(rev, path)` on `VcsToolkit.Core` — the content of a file as it exists at a revision, untrimmed, dispatched over `Git.ShowFile`/`Jj.FileShow` (`rev` is a git commit-ish or a jj revset, passed through as-is) — and the MCP `repo_show_file` read tool exposing it.
- Hermetic conflict-marker models (pure, no subprocess) for reading a conflicted file's content into typed regions and writing a chosen resolution back. `VcsToolkit.Git.Conflict` handles git's `merge`/`diff3`/`zdiff3` styles (`ResolutionSide`, `ConflictRegion`, `ConflictSegment`, `hasConflictMarkers`/`parseConflicts`/`render`/`resolve`), and `VcsToolkit.Jj.Conflict` handles jj's native `diff`/`snapshot` materialized styles (`JjConflictSection`, `JjConflictRegion` with `Sides`/`Base`, `JjConflictSegment`, `JjResolution`, and the same function set) — a jj file materialized with jj's `git` style is redirected to the git model. Both `render` is a byte-exact roundtrip of `parseConflicts` (verbatim endings, CRLF, and a missing trailing newline preserved), and `resolve` honors jj's missing-terminating-newline encoding so a written-back resolution is byte-exact. Drive them from the already-exposed conflicted-path listers (`Repo.ConflictedFiles`, `Git.ConflictedFiles`, `Jj.ResolveList`) plus `Git.ShowFile` / `Jj.FileShow`.
- Directory-bound `at(dir)` views on every client and the `Repo` facade — an ergonomic layer over the `dir`-taking methods for code that works in one directory. `Git.At(dir)` / `Jj.At(dir)` / `GitHub.At(dir)` / `GitLab.At(dir)` / `Gitea.At(dir)` return a `GitAt` / `JjAt` / `GitHubAt` / `GitLabAt` / `GiteaAt` whose modelled methods drop the leading `dir` argument (so `git.At(dir).Status()` is exactly `git.Status dir`, byte-identical argv), while the raw `Run`/`RunRaw` escape hatches deliberately stay bound to the process's current directory (not `dir`). `JjAt.Transaction` forwards the op-log-rollback scope with `dir` pre-bound. `Repo.GitAt` / `Repo.JjAt` hand out the underlying client view bound to the handle's `Cwd` (`Some` only for the matching backend); re-anchor with `repo.At(path)` first to bind another worktree.
- `VcsToolkit.Mcp` + the `vcs-mcp` binary (`VcsToolkit.Mcp.Server`): a Model Context Protocol server exposing the toolkit's typed git/jj + forge operations as agent-callable tools, ported from the Rust `vcs-mcp` crate. The `VcsToolkit.Mcp` library holds the hermetically-testable core: `VcsMcpServer` with 32 tools — the `repo_*` group (`repo_snapshot`/`repo_info`/`repo_status`/`repo_diff_stat`/`repo_branches`/`repo_current_branch`/`repo_conflicts`/`repo_worktrees` reads; `repo_try_merge`/`repo_commit`/`repo_checkout`/`repo_fetch`/`repo_push`/`repo_create_worktree`/`repo_remove_worktree` writes) over `VcsToolkit.Core.Repo`, and the `forge_*` group (`forge_auth_status`/`forge_repo_view`/`forge_info`/`forge_pr_list`/`forge_pr_view`/`forge_pr_checks`/`forge_issue_list`/`forge_issue_view`/`forge_release_list`/`forge_release_view` reads; `forge_issue_create`/`forge_pr_create`/`forge_pr_merge`/`forge_pr_close`/`forge_pr_mark_ready`/`forge_pr_comment`/`forge_pr_edit` writes) over `VcsToolkit.Forge.Forge`. Read tools are always available; the 14 mutating tools are gated by a `WriteGate` (`None`/`All`/`Set`) driven by `--allow-write`/`--allow-tools`, and repo mutations serialize on a per-server lock. Also the tool catalogue with generated JSON-Schema input schemas and its tool dispatcher, an `McpError` with `coreErr`/`forgeErr`/`guardArgvField` mapping onto MCP's invalid-params/internal-error codes, and the CLI argument parser (`--repo`/`--forge`/`--allow-write`/`--allow-tools`/`--timeout`/`--help`). The `vcs-mcp` binary serves over stdio via the `ModelContextProtocol` SDK, auto-detects the forge from the `origin` remote (or `--forge`), and uses a hardened, timeout-bound git client. (JSON output is camelCase; the `VcsToolkit.Mcp` layer adds `FSharp.SystemTextJson`, and the binary adds `ModelContextProtocol` + `Microsoft.Extensions.Hosting`.)
- `VcsToolkit.Watch`: filesystem-watch a git/jj repository and emit typed state-change events, ported from the Rust `vcs-watch` crate — a `RepoWatcher` (`Watch`/`Builder` with `WorkingTree`/`Debounce`/`MaxWait`/`RequeryTimeout`) watches the `.git`/`.jj` state dir (resolving worktree gitlinks + the shared `commondir`), debounces the write burst (with a `MaxWait` ceiling), re-queries `Repo.Snapshot` + `LocalBranches` (bounded by `RequeryTimeout`), and diffs against the previous state to yield `RepoChange` (fresh snapshot + non-empty `RepoEvent` list). `RepoEvent` covers `HeadMoved`/`BranchSwitched`/`BranchCreated`/`BranchDeleted`/`WorkingCopyChanged`/`UpstreamChanged`/`AheadBehindChanged`/`OperationChanged`/`ConflictChanged`. Pull changes with `Recv`, read the last snapshot via `Current`, and monitor health via `Stats` (`WatcherStats`: requeries/changes/skips/`LastError`/watch-errors). A `WatchError` (`Notify`/`Vcs`/`Io`) carries `IsTransient`/`IsNotFound`/`ProcessError`. Built on `System.IO.FileSystemWatcher` + `System.Threading.Channels` (the .NET analogues of the Rust `notify` + tokio stack); `Dispose` stops the watch and the background loop.
- `VcsToolkit.TestKit`: throwaway git/jj sandboxes for integration tests, ported from the Rust `vcs-testkit` crate — a self-cleaning `TempDir` (unique via pid + a process-wide counter, removed on `Dispose`), `GitSandbox` (`Init`/`Write`/`AddAll`/`Commit`/`CommitFile`/`Branch`/`Checkout`/`RevParse`), `JjSandbox` (`Init`/`Write`/`Describe`/`NewChange`/`Bookmark`), a seeded `BareRemote` (a local clone/fetch/push source), and the raw `Raw.git`/`Raw.jj`/`Raw.configureIdentity` steps for directories no sandbox owns. Dependency-free (references no other `VcsToolkit` package, so it can be a test dependency of any of them without a cycle), hermetic (every git/jj invocation is isolated from the host's VCS config), and raising on failure. Runs the actual `git`/`jj` on `PATH`.
- `VcsToolkit.CliSupport`: shared plumbing ported from the Rust `vcs-cli-support` crate — the `rejectFlagLike` argv injection guard, error classifiers (`isMergeConflict`, `isNothingToCommit`, `isTransientFetchError`, `isLockContention`), the `RetryPolicy` / `Retry.retryAsync` lock-contention retry, credential provisioning (`Secret`, `Credential`, `ICredentialProvider`, `StaticCredential`, `EnvToken`, `Credentials.gitCredentialHelper`), the `ManagedClient` wrapper over the ProcessKit runner, and the tolerant, total `Json` helpers (`Json.strOr`/`strOpt`/`strFirst`/`u64Or`/`boolOr`/`nestedStr`/`arrayOf`/`withDoc`/`parseArray`/`parseArrayResult`/`parseObject`) that the forge wrappers build their `--json` parsers on.
- `VcsToolkit.Diff`: the git-format unified-diff model (`FileDiff`, `Hunk`, `DiffLine`, `ChangeKind`, `DiffStat`) and `parseDiff` parser, plus `Version` and `parseDottedVersion`, ported from the Rust `vcs-diff` crate.
- `VcsToolkit.Git`: the `git` CLI client (`Git`) ported from the Rust `vcs-git` crate — status (porcelain v1/v2), branches, commit/`commitPaths`, checkout, diff/log, merge/rebase/reset, fetch/push/clone, worktrees, tags, blame, config, remotes, cherry-pick/revert, `switchWithStash`, and a `harden` profile — with the builder specs (`CommitPaths`, `GitPush`, `MergeCommit`, `MergeNoCommit`, `WorktreeAdd`, `CloneSpec`, `AnnotatedTag`), validated `RefName`/`RevSpec` newtypes, `GitCapabilities`, and the git output parsers.
- `VcsToolkit.Jj`: the Jujutsu (`jj`) CLI client (`Jj`) ported from the Rust `vcs-jj` crate — changes/`log`, `describe`/`new`, bookmarks (local and remote-tracking), the operation log (`opLog`/`opRestore`/`opUndo`) with op-log-rollback `transaction`s, workspaces (incl. a bounded `workspaceRoots` fan-out), `squash`/`split`/`absorb`/`duplicate`/`abandon`, diff and template queries, and git sync (`gitFetch`/`gitPush`/`gitClone`/`gitImport`) — with the builder specs (`WorkspaceAdd`, `SquashPaths`), `JjFileset`/`RevsetExpr` newtypes, `JjCapabilities` gating on the validated jj >= 0.38 floor, and the jj output parsers.
- `VcsToolkit.GitHub`: the GitHub (`gh`) CLI client (`GitHub`) ported from the Rust `vcs-github` crate — pull requests (`prList`/`prView`/`prCreate`/`prMerge`/`prMarkReady`/`prClose`/`prReview`/`prComment`/`prEdit`/`prChecks`/`prFeedback`), issues (`issueList`/`issueCreate`/`issueView`), Actions runs (`runList`/`runView`/`runWatch`), releases (`releaseList`/`releaseView`), `repoView`, `authStatus`, and the raw REST/GraphQL `api` escape hatch — with the option types (`PrCreate`, `PrEdit`, `PrMerge`/`MergeStrategy`, `ReviewAction`/`ReviewKind`), the `--json` DTOs (`PullRequest`, `Issue`, `WorkflowRun`, `CheckRun`/`CheckBucket`, `Release`, `Review`/`Comment`/`PrFeedback`, `Repo`), and tolerant `System.Text.Json` parsers. Tokens resolve to `GH_TOKEN` (never argv).
- `VcsToolkit.GitLab`: the GitLab (`glab`) CLI client (`GitLab`) ported from the Rust `vcs-gitlab` crate — the lean merge-request lifecycle (`mrList`/`mrView`/`mrCreate`/`mrMerge`/`mrMarkReady`/`mrClose`/`mrComment`/`mrEdit`), CI/pipeline status (`mrChecks` → `CiStatus`), issues (`issueList`/`issueView`/`issueCreate`), releases (`releaseList`/`releaseView`), `repoView`, `authStatus`, and the REST/GraphQL `api` escape hatch — with the option types (`MrCreate`, `MrEdit`, `MergeStrategy`), the `--output json` DTOs (`MergeRequest`, `Repo`, `Issue`, `Release`, `CiStatus`), and tolerant parsers. `mrMerge` merges immediately (`--auto-merge=false`); tokens resolve to `GITLAB_TOKEN` (never argv).
- `VcsToolkit.Forge`: the unified forge facade ported from the Rust `vcs-forge` crate — one `Forge` handle (`GitHub`/`GitLab`/`Gitea`/`GitHubWithToken`/`GitLabWithToken`/`FromGitHub`/`FromGitLab`/`FromGitea`/`FromUnknown`) dispatches the shared PR/MR lifecycle (`AuthStatus`/`RepoView`/`PrList`/`PrView`/`PrCreate`/`PrComment`/`PrEdit`/`PrMerge`/`PrMarkReady`/`PrClose`/`PrChecks`, `Capabilities`, `IssueList`/`IssueView`/`IssueCreate`, `ReleaseList`/`ReleaseView`) to the underlying `VcsToolkit.GitHub` / `VcsToolkit.GitLab` / `VcsToolkit.Gitea` client — with the unified DTOs (`ForgeKind` + a security-hardened `OfRemoteUrl` host classifier, `ForgePr`/`ForgePrState`, `ForgeRepo`, `ForgeIssue`/`ForgeIssueState`, `ForgeRelease`, `CiStatus`, `PrCreate`, `MergeStrategy`, `PrEdit`, `ForgeCapabilities`, `ForgeOp`) normalising the three CLIs' shapes, and a `ForgeError` (`Forge`/`Unsupported`/`InvalidInput`) with `IsUnauthorized`/`IsRateLimited`/`IsTransientFetchError`/`IsTransient`/`IsNotFound`/`IsUnsupported` classifiers. A few ops (`RepoView`/`PrMarkReady`/`PrChecks`/`ReleaseView`) are `Unsupported` on Gitea; `Supports(ForgeOp)` branches before calling. The raw backend client is reachable via `GitHubClient`/`GitLabClient`/`GiteaClient` (`Some` only for the matching backend) for the per-forge operations off the shared surface — the `Repo.Git`/`Repo.Jj` escape-hatch analogue.
- `VcsToolkit.Core`: the backend-agnostic `Repo` facade ported from the Rust `vcs-core` crate — `Repo.Open` auto-detects git vs jj (`Detect.detect`: a pure `.git`/`.jj` upward probe, jj winning when colocated), then dispatches one common surface (`CurrentBranch`/`Trunk`/`LocalBranches`/`BranchExists`/`DeleteBranch`/`RenameBranch`, `ChangedFiles`/`DiffStat`/`HasUncommittedChanges`/`HasTrackedChanges`/`ConflictedFiles`/`Snapshot`, `CommitPaths`/`Fetch`/`FetchFrom`/`FetchBranch`/`Push`/`Checkout`/`Rebase`, `TryMerge`/`InProgressState`/`AbortInProgress`/`ContinueInProgress`, `ListWorktrees`/`CreateWorktree`/`RemoveWorktree`) to the underlying `VcsToolkit.Git` / `VcsToolkit.Jj` client — with the backend-agnostic DTOs (`BackendKind`, `FileChange`, `WorktreeInfo`, `OperationState`, `UpstreamTracking`, `RepoSnapshot`, `MergeProbe`, `CreateOutcome`) reusing `VcsToolkit.Diff`'s `ChangeKind`/`DiffStat`, and a `RepoError` (`NotARepository`/`WorktreeNotFound`/`Io`/`Vcs`) with `IsMergeConflict`/`IsNothingToCommit`/`IsLockContention`/`IsTransientFetchError`/`IsTransient`/`IsNotFound` classifiers. `Repo.FromGit`/`FromJj` are the test seam; `.Git`/`.Jj` are the raw-client escape hatches. The synchronous `cleanupWorktreeBlocking` Drop-guard helper is not ported (.NET's `IAsyncDisposable` can `await RemoveWorktree`).
- `VcsToolkit.Gitea`: the Gitea/Forgejo (`tea`) CLI client (`Gitea`) ported from the Rust `vcs-gitea` crate — the lean pull-request lifecycle (`prList`/`prView`/`prCreate`/`prMerge`/`prClose`/`prComment`/`prEdit`), issues (`issueList`/`issueView`/`issueCreate`), and release listing (`releaseList`) — with the option types (`PrCreate`, `PrEdit`, `MergeStrategy`), the DTOs (`PullRequest`, `Issue`, `Release`) parsed from tea's all-strings print-table (and the typed issue-detail object), and `authStatus` via the configured logins. Authentication is ambient (`tea`'s stored logins); `prView` is synthesized by listing and filtering.
- Forge CLI version parsing + `Capabilities` on the three forge wrappers: `GitHub.Capabilities()` / `GitLab.Capabilities()` / `Gitea.Capabilities()` probe `gh`/`glab`/`tea --version` and return a `GitHubCapabilities` / `GitLabCapabilities` / `GiteaCapabilities` carrying the parsed `VcsToolkit.Diff.Version`, an `IsSupported` floor check, `EnsureSupported()`, and the static `MinimumSupported` floor (gh ≥ 2.0, glab ≥ 1.0, tea ≥ 0.9). An unrecognisable `--version` banner degrades to a predictable `Parse` error, never a throw. (The three wrappers now also depend on `VcsToolkit.Diff` for the shared `Version` type.)
- `VcsToolkit.Forge`: `Forge.Capabilities()` now reports the detected CLI `Version` and backend `Kind` on `ForgeCapabilities` (surfaced independently of `Authed`, so the version shows even when unauthenticated; the `Unknown` handle reports `Version = None` / `Kind = Unknown`), and the mutating operations `PrCreate`/`PrMerge`/`PrEdit`/`IssueCreate` are version-gated — refused up front with the new structural `ForgeError.UnsupportedVersion` (carrying the forge, operation, found, and minimum versions) when the installed CLI is confirmed below the wrapper's floor, instead of failing on a raw CLI error. A version that can't be probed or parsed falls through (fail-open), and read-only operations are never gated.
- Packaging: every published package ships a NuGet package icon, and each package declares its inter-package `VcsToolkit.*` dependencies (so restoring one package pulls in the siblings it needs).
- `vcs-mcp`'s `repo_show_file` truncates content past a configurable output budget (`--output-budget <bytes>`, default 200000; `0` disables the cap) with an explicit `[truncated: showing N of M bytes]` marker, so a huge file read can no longer blow out an agent's context; content within the budget is returned byte-for-byte unchanged.
- `Git.EmptyTreeOid(dir)` (and `GitAt.EmptyTreeOid()`) — the empty-tree object id in a repository's actual object format, resolved via `git hash-object -t tree --stdin` fed empty input (nothing is written to the object database).
- Locally check out a pull/merge request's branch: `GitHub.PrCheckout` (`gh pr checkout`), `GitLab.MrCheckout` (`glab mr checkout`), `Gitea.PrCheckout` (`tea pr checkout`), the unified `Forge.PrCheckout` (dispatched across all three; `Unsupported` only on the CLI-less `Unknown` handle), and the write-gated `vcs-mcp` `forge_pr_checkout` tool. Unlike the remote-only forge operations this switches the local working tree to the PR/MR's source branch, so on the MCP server it serializes with the `repo_*` mutations.
- `VcsToolkit.Core`'s `OperationState` now models the remaining git sequencer states — `CherryPick` (`CHERRY_PICK_HEAD`), `Revert` (`REVERT_HEAD`), and `Bisect` (`BISECT_LOG`) — alongside `Merge`/`Rebase`/`ApplyMailbox`. `Repo.Snapshot`/`Repo.InProgressState` now detect them (keyed off each operation's own git-dir marker, so a cherry-pick/revert conflict is no longer mis-read as `Clear`), `Repo.AbortInProgress` routes `cherry-pick --abort` / `revert --abort` / `bisect reset`, and `Repo.ContinueInProgress` routes `cherry-pick --continue` / `revert --continue` (a re-conflict on the next commit surfaces as `Conflict`, not an error). A bisect has no continue step, so `Repo.ContinueInProgress` during a bisect is refused up front with the new `RepoError.Unsupported` (test with `IsUnsupported`) instead of silently doing nothing. The underlying `VcsToolkit.Git` client gains the matching probes (`IsCherryPickInProgress`/`IsRevertInProgress`/`IsBisectInProgress`) and commands (`CherryPickAbort`/`CherryPickContinue`/`RevertAbort`/`RevertContinue`/`BisectReset`), on both `Git` and the `GitAt` view. `VcsToolkit.Watch`'s `RepoEvent.OperationChanged` now reports transitions through these states too.
- `Jj.RollbackTo(dir, capturedOpHead)` (and `JjAt.RollbackTo(capturedOpHead)`) on `VcsToolkit.Jj` — the shared op-log rollback protocol behind `Jj.Transaction` and `Repo.TryMerge`, exposed for direct use. It runs its cleanup (`op log` + `op restore`) on a fresh cancellation token with its own timeout (never inheriting an already-cancelled operation's budget) and, before restoring, probes the recent op log for divergence: if a concurrent operation advanced the op-head past `capturedOpHead`, it refuses to restore rather than clobbering that work. The outcome is reported via the new `RollbackOutcome` type (`RolledBack` / `SkippedDiverged of captured * current`).
- `VcsToolkit.Forge`: `ForgePr` and `ForgeIssue` gain `Labels` and `Assignees` (`string list option`) — GitHub and GitLab report a confirmed `Some` (an empty `Some []` is a confirmed "no labels" / "unassigned"), while Gitea is always `None` (its `tea` CLI has no labels/assignees columns, so an empty list would be a false "none" rather than the honest "unknown"), mirroring the `None`/`Some []` contract of `ForgePr.Draft`. The underlying `VcsToolkit.GitHub` and `VcsToolkit.GitLab` `PullRequest`/`Issue`/`MergeRequest` DTOs also carry the flattened `Labels`/`Assignees` lists (`gh`'s `--json` field set now requests `labels,assignees`; `glab`'s `--output json` already returns them).
- Typed pull/merge request diffs: `GitHub.PrDiff(dir, number)` (`gh pr diff <n>`) and `GitLab.MrDiff(dir, number)` (`glab mr diff <n>`) return the PR/MR's unified diff parsed into per-file `VcsToolkit.Diff.FileDiff` values, and the unified `Forge.PrDiff(number)` dispatches to the matching backend — `Unsupported` on Gitea (`tea` has no diff command) and on the CLI-less `Unknown` handle.

### Changed
- `VcsToolkit.CliSupport`/`VcsToolkit.Git` now scope credential resolution to the operation's known host so a multi-host `ICredentialProvider` serves the right secret. `ManagedClient` gains `WithExpectedHost(host)`, whose value becomes the `CredentialRequest.Host` on the token-env injection path (previously always `None`); and `git`'s remote operations pass the clone URL's host into the provider lookup (previously ignored — the host scoped only the credential helper, not the secret selection), so the one host drives both provider selection and the helper's host gate and they cannot disagree. The fallback policy is unchanged but now documented and covered for reads and writes: no provider / provider `Ok None` / an empty secret all defer to ambient auth, while a provider `Error` fails closed (the operation aborts instead of silently running unauthenticated).
- `VcsToolkit.Jj`'s `Jj.Transaction` rollback (and the jj path of `VcsToolkit.Core`'s `Repo.TryMerge`) is now hardened against cancellation and concurrency: the on-error / post-probe cleanup runs on a fresh cancellation budget, so a cancelled or timed-out closure is still rolled back instead of silently skipping restore, and a divergence probe refuses to `op restore` when a concurrent operation advanced the op-head past the captured point (for `TryMerge`, a diverged rollback surfaces as an error rather than a `MergeProbe` result that would misdescribe the tree). `Jj.Transaction` still returns the closure's original error unchanged.
- `VcsToolkit.Forge`: `Forge.PrMerge` now takes a unified `PrMerge` merge-spec (`Strategy`/`Auto`/`DeleteBranch`, built via `PrMerge.Merge`/`Squash`/`Rebase` + `WithAuto`/`WithDeleteBranch`) instead of a bare `MergeStrategy`, so auto-merge and delete-source-branch are available off the forge-agnostic surface. On GitHub `Auto`/`DeleteBranch` map to gh's real `--auto`/`--delete-branch` flags; on GitLab and Gitea — whose CLIs expose no confirmed equivalent — a spec asking for either is refused with `ForgeError.Unsupported` before any spawn rather than silently dropping the option, while a plain strategy merge still works on all three. The `vcs-mcp` `forge_pr_merge` tool gains matching optional `auto`/`delete_branch` boolean parameters.
- `VcsToolkit.Forge`: the best-effort `ForgePr.Draft`, `ForgeRepo.Private`, `ForgeRelease.Draft`, and `ForgeRelease.Prerelease` fields change type from `bool` to `bool option`. `Some` now carries a value the backend actually reported and `None` means it does not surface the field, so an unreported draft/visibility is no longer collapsed into a confirmed `false`: `None` for a GitHub/Gitea PR draft, a GitLab release draft/pre-release, and an absent GitLab project visibility; `Some` elsewhere.
- `VcsToolkit.CliSupport`'s `Retry.retryAsync` now takes a `CancellationToken` (a new required parameter between `shouldRetry` and `op`) and observes it around the backoff sleep between attempts — a cancellation firing mid-backoff aborts the retry immediately (instead of sleeping out the remaining delay) and does not start another attempt, surfacing as `ProcessError.Cancelled`; `ManagedClient`'s retry paths now propagate their configured cancellation token through this backoff.
- `VcsToolkit.Watch`'s `RepoWatcher` background loop now retries a **transient** re-query failure (per `WatchError.IsTransient`) in place with a bounded, cancellable backoff instead of silently skipping it every debounce cycle, and treats a **terminal** failure (not transient, or the retry budget exhausted) as a hard stop: it closes the output channel with an error (`WatcherTerminated`, carrying the underlying `WatchError`) instead of skip-and-continue forever. `RepoWatcher.Recv()` now re-raises that error (as `ChannelClosedException.InnerException`) instead of returning `None`, so a consumer can tell "the watch broke" apart from an ordinary `Dispose` (which still yields `None`).
- `VcsToolkit.Mcp`'s `VcsMcpServer` constructor gains a fourth `outputBudget: int option` parameter (the output-size budget, in bytes, applied to `repo_show_file`; `None` means no limit) — a breaking change to the constructor's arity.

### Fixed
- `Git.Add`/`Git.CommitPaths` now apply `--literal-pathspecs` unconditionally, so a path containing a glob metacharacter (`*`, `?`, `[]`) matches literally instead of being silently expanded as a pathspec pattern; a path set whose combined argv length would approach the OS command-line limit (Windows' ~32767-character `CreateProcess` ceiling) is now routed through the NUL-safe `--pathspec-from-file=- --pathspec-file-nul` stdin transport instead of risking a spawn failure, with a small path set still going through a single, argv-unchanged call.
- `GitLab`'s `MrCreate`/`MrEdit`/`IssueCreate`/`MrComment` now refuse a body/description of exactly `"-"` before spawning `glab`, instead of hanging on glab's own stdin/`$EDITOR` sentinel prompt.
- `Git.Upstream` no longer collapses a detached `HEAD` or a directory outside a repository into a false "no upstream configured" (`Ok None`); it now checks that `HEAD` is an attached branch first and surfaces those cases (and any `rev-parse @{u}` failure other than the documented "no upstream" exit code) as `Error`.
- `VcsToolkit.Jj`'s `JjFileset.Path` no longer unconditionally rewrites `\` to `/`: on non-Windows platforms a backslash is a legitimate filename byte, so it is now preserved and escaped as `\\` (instead of being silently turned into `/`, mismatching the real file, or left unbalanced at the end of the path); the `\`→`/` normalisation still applies only on Windows.
- `Jj.Status`/`Jj.DiffSummary` now resolve the workspace root and run the underlying `jj diff --summary` FROM it, instead of from whichever `dir` the caller bound — so the returned paths (including `OldPath` for a rename/copy) are workspace-root-relative regardless of which subdirectory `dir` names, matching the facade's repo-relative promise for the same DTO on the git backend, instead of returning cwd-relative paths (which could carry a literal `..` when called from a subdirectory). A reported path that still escapes the workspace root is now rejected as an error rather than silently propagated.
- `Git.DiffText`/`Diff` diffing an unborn (no-commits-yet) working tree no longer targets the hardcoded SHA-1 empty-tree constant, which does not exist as an object in a repository under `extensions.objectFormat=sha256`; it now resolves the empty-tree id via the new `Git.EmptyTreeOid`, which asks the repository's own `git hash-object` for it.
