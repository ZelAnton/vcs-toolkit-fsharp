# Changelog

All notable changes to **VcsToolkit** are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added
- List a repository's configured remotes: `Repo.Remotes()` on `VcsToolkit.Core` returns a backend-agnostic `Remote` (name + URL) list, dispatched over the new typed `Git.Remotes(dir)` (`git remote -v`, deduplicated to one entry per remote carrying its fetch URL) and `Jj.GitRemoteList(dir)` (`jj git remote list`, run with `--ignore-working-copy` + `--color never`), plus the `GitAt.Remotes()`/`JjAt.GitRemoteList()` bound-view forwarders and the read-only `vcs-mcp` `repo_remotes` tool. `vcs-mcp`'s forge auto-detection now uses the typed `Jj.GitRemoteList` instead of hand-parsing `jj git remote list` output.
- Create a release across all three forges: `Forge.ReleaseCreate(spec)` (dispatched to `gh`/`glab`/`tea release create`) with a unified `ReleaseCreate` spec (tag + optional title/notes + draft/prerelease), the per-client `GitHub.ReleaseCreate`/`GitLab.ReleaseCreate`/`Gitea.ReleaseCreate` methods (and their `At`-view forwarders, plus each client's own `ReleaseCreate` option type — GitLab's omits draft/pre-release), and the write-gated `vcs-mcp` `forge_release_create` tool. `Draft`/`Prerelease` map to real `gh`/`tea` flags on GitHub and Gitea; GitLab's `glab` has no release draft/pre-release concept, so a spec asking for either is refused with `Unsupported` before any spawn — query it up front via the new `Forge.SupportsReleaseOptions`. Version-gated like the other facade mutations, and (on GitHub) `--notes` is always emitted so `gh release create` never falls back to an interactive editor prompt.
- A generated documentation site (fsdocs/FSharp.Formatting), published to GitHub Pages on push to `main` and on release — the API reference for every publishable `src/` package plus the architecture, examples, and MCP-server guides as content pages. See the link in the README.
- `Repo.Annotate(path, rev)` on `VcsToolkit.Core` — per-line authorship of a file ("who last touched this line, and when"), dispatched over `Git.Blame`/`Jj.FileAnnotate` into a unified `AnnotateLine` DTO (line number, content, commit/change id, and author/date), and the MCP `repo_annotate` read tool exposing it (its JSON output is truncated to the same `--output-budget` as `repo_show_file`). `jj file annotate`'s template was extended (`ANNOTATE_TEMPLATE`, `VcsToolkit.Jj.AnnotationLine` gains `Author`/`Time`) to surface authorship/timestamp — unlike the typed `Repo.Log`, whose `Author`/`Date` stay `None` on jj because its log template doesn't ask jj for them, `Repo.Annotate`'s `Author`/`Date` are filled on both backends.
- `Repo.Clone(url, dest, spec)` / `Repo.CloneWith(url, dest, spec, git, jj)` on `VcsToolkit.Core` — a backend-agnostic "URL → open `Repo` handle" entry point for provisioning flows, with a `CloneOptions`/`CloneKind` spec selecting `Git` / `JjColocated` / `JjNonColocated` and delegating to `Git.CloneRepo`/`Jj.GitClone` for the actual clone, then `OpenWith` to build the handle (same path normalisation and `RepoError` mapping as `Open`/`OpenWith`).
- XML documentation generation for every `src/` library, packaged into the `.nupkg` alongside the assembly (`lib/<tfm>/<Assembly>.xml`) — IntelliSense/IDE tooltips now surface the full `///`-documented public API for consumers of the NuGet packages.
- RepoWatcher.ReadAll(?cancellationToken) on VcsToolkit.Watch — consume repository changes as an IAsyncEnumerable<RepoChange> for idiomatic await foreach usage, with Current advancing for each yielded change.
- `Repo.NewChild(reference)` on `VcsToolkit.Core` — start new work on top of `reference` without modifying it (git `checkout <reference> --`; jj `new <reference>`, via the new `Jj.NewChild`/`JjAt.NewChild`). On jj this differs from `Checkout`/`jj edit`: it stacks a fresh, undescribed child change on `reference` instead of rewriting `reference`'s own commit in place.
- `Repo.ShowFile(rev, path)` on `VcsToolkit.Core` — the content of a file as it exists at a revision, untrimmed, dispatched over `Git.ShowFile`/`Jj.FileShow` (`rev` is a git commit-ish or a jj revset, passed through as-is) — and the MCP `repo_show_file` read tool exposing it. The string form is UTF-8-decoded, so it is byte-exact only for text: a non-UTF-8 byte (a binary or legacy-encoded blob) is replaced with U+FFFD and does not round-trip — use the `*Bytes` variants below for verbatim binary content.
- `Repo.ShowFileBytes(rev, path)` on `VcsToolkit.Core`, with `Git.ShowFileBytes`/`Jj.FileShowBytes` and the `GitAt`/`JjAt` views — a file's content at a revision as raw, verbatim `byte[]` (built on `ManagedClient.OutputBytes`), so arbitrary binary/legacy-encoded (non-UTF-8) blob content round-trips byte-for-byte for a genuinely byte-exact read-modify-write, where the UTF-8-decoding string `ShowFile`/`FileShow` would silently corrupt it to U+FFFD.
- Hermetic conflict-marker models (pure, no subprocess) for reading a conflicted file's content into typed regions and writing a chosen resolution back. `VcsToolkit.Git.Conflict` handles git's `merge`/`diff3`/`zdiff3` styles (`ResolutionSide`, `ConflictRegion`, `ConflictSegment`, `hasConflictMarkers`/`parseConflicts`/`render`/`resolve`), and `VcsToolkit.Jj.Conflict` handles jj's native `diff`/`snapshot` materialized styles (`JjConflictSection`, `JjConflictRegion` with `Sides`/`Base`, `JjConflictSegment`, `JjResolution`, and the same function set) — a jj file materialized with jj's `git` style is redirected to the git model. Both `render` is a byte-exact roundtrip of `parseConflicts` (verbatim endings, CRLF, and a missing trailing newline preserved), and `resolve` honors jj's missing-terminating-newline encoding so a written-back resolution is byte-exact. Drive them from the already-exposed conflicted-path listers (`Repo.ConflictedFiles`, `Git.ConflictedFiles`, `Jj.ResolveList`) plus `Git.ShowFile` / `Jj.FileShow`.
- Directory-bound `at(dir)` views on every client and the `Repo` facade — an ergonomic layer over the `dir`-taking methods for code that works in one directory. `Git.At(dir)` / `Jj.At(dir)` / `GitHub.At(dir)` / `GitLab.At(dir)` / `Gitea.At(dir)` return a `GitAt` / `JjAt` / `GitHubAt` / `GitLabAt` / `GiteaAt` whose modelled methods drop the leading `dir` argument (so `git.At(dir).Status()` is exactly `git.Status dir`, byte-identical argv), and whose raw `Run`/`RunRaw` escape hatches run in the bound `dir` too (the process-cwd form stays on the unbound client). `JjAt.Transaction` forwards the op-log-rollback scope with `dir` pre-bound. `Repo.GitAt` / `Repo.JjAt` hand out the underlying client view bound to the handle's `Cwd` (`Some` only for the matching backend); re-anchor with `repo.At(path)` first to bind another worktree.
- `VcsToolkit.Mcp` + the `vcs-mcp` binary (`VcsToolkit.Mcp.Server`): a Model Context Protocol server exposing the toolkit's typed git/jj + forge operations as agent-callable tools, ported from the Rust `vcs-mcp` crate. The `VcsToolkit.Mcp` library holds the hermetically-testable core: `VcsMcpServer` with 32 tools — the `repo_*` group (`repo_snapshot`/`repo_info`/`repo_status`/`repo_diff_stat`/`repo_branches`/`repo_current_branch`/`repo_conflicts`/`repo_worktrees` reads; `repo_try_merge`/`repo_commit`/`repo_checkout`/`repo_fetch`/`repo_push`/`repo_create_worktree`/`repo_remove_worktree` writes) over `VcsToolkit.Core.Repo`, and the `forge_*` group (`forge_auth_status`/`forge_repo_view`/`forge_info`/`forge_pr_list`/`forge_pr_view`/`forge_pr_checks`/`forge_issue_list`/`forge_issue_view`/`forge_release_list`/`forge_release_view` reads; `forge_issue_create`/`forge_pr_create`/`forge_pr_merge`/`forge_pr_close`/`forge_pr_mark_ready`/`forge_pr_comment`/`forge_pr_edit` writes) over `VcsToolkit.Forge.Forge`. Read tools are always available; the 14 mutating tools are gated by a `WriteGate` (`None`/`All`/`Set`) driven by `--allow-write`/`--allow-tools`, and repo mutations serialize on a per-server lock. Also the tool catalogue with generated JSON-Schema input schemas and its tool dispatcher, an `McpError` with `coreErr`/`forgeErr`/`guardArgvField` mapping onto MCP's invalid-params/internal-error codes, and the CLI argument parser (`--repo`/`--forge`/`--allow-write`/`--allow-tools`/`--timeout`/`--help`). The `vcs-mcp` binary serves over stdio via the `ModelContextProtocol` SDK, auto-detects the forge from the `origin` remote (or `--forge`), and uses a hardened, timeout-bound git client. (JSON output is camelCase; the `VcsToolkit.Mcp` layer adds `FSharp.SystemTextJson`, and the binary adds `ModelContextProtocol` + `Microsoft.Extensions.Hosting`.)
- `VcsToolkit.Watch`: filesystem-watch a git/jj repository and emit typed state-change events, ported from the Rust `vcs-watch` crate — a `RepoWatcher` (`Watch`/`Builder` with `WorkingTree`/`Debounce`/`MaxWait`/`RequeryTimeout`) watches the `.git`/`.jj` state dir (resolving worktree gitlinks + the shared `commondir`), debounces the write burst (with a `MaxWait` ceiling), re-queries `Repo.Snapshot` + `LocalBranches` (bounded by `RequeryTimeout`), and diffs against the previous state to yield `RepoChange` (fresh snapshot + non-empty `RepoEvent` list). `RepoEvent` covers `HeadMoved`/`BranchSwitched`/`BranchCreated`/`BranchDeleted`/`WorkingCopyChanged`/`UpstreamChanged`/`AheadBehindChanged`/`OperationChanged`/`ConflictChanged`. Pull changes with `Recv`, read the last snapshot via `Current`, and monitor health via `Stats` (`WatcherStats`: requeries/changes/skips/`LastError`/watch-errors). A `WatchError` (`Notify`/`Vcs`/`Io`) carries `IsTransient`/`IsNotFound`/`ProcessError`. Built on `System.IO.FileSystemWatcher` + `System.Threading.Channels` (the .NET analogues of the Rust `notify` + tokio stack); `Dispose` stops the watch and the background loop.
- `VcsToolkit.TestKit`: throwaway git/jj sandboxes for integration tests, ported from the Rust `vcs-testkit` crate — a self-cleaning `TempDir` (unique via pid + a process-wide counter, removed on `Dispose`), `GitSandbox` (`Init`/`Write`/`AddAll`/`Commit`/`CommitFile`/`Branch`/`Checkout`/`RevParse`), `JjSandbox` (`Init`/`InitNonColocated`/`Write`/`Describe`/`NewChange`/`Bookmark`), a seeded `BareRemote` (a local clone/fetch/push source), and the raw `Raw.git`/`Raw.jj`/`Raw.configureIdentity` steps for directories no sandbox owns. Dependency-free (references no other `VcsToolkit` package, so it can be a test dependency of any of them without a cycle), hermetic (every git/jj invocation is isolated from the host's VCS config), and raising on failure. Runs the actual `git`/`jj` on `PATH`.
- `VcsToolkit.CliSupport`: shared plumbing ported from the Rust `vcs-cli-support` crate — the `rejectFlagLike` argv injection guard, error classifiers (`isMergeConflict`, `isNothingToCommit`, `isTransientFetchError`, `isLockContention`), the `RetryPolicy` / `Retry.retryAsync` lock-contention retry, credential provisioning (`Secret`, `Credential`, `ICredentialProvider`, `StaticCredential`, `EnvToken`, `Credentials.gitCredentialHelper`), the `ManagedClient` wrapper over the ProcessKit runner, and the tolerant, total `Json` helpers (`Json.strOr`/`strOpt`/`strFirst`/`u64Or`/`boolOr`/`nestedStr`/`arrayOf`/`withDoc`/`parseArray`/`parseArrayResult`/`parseObject`) that the forge wrappers build their `--json` parsers on.
- `VcsToolkit.Diff`: the git-format unified-diff model (`FileDiff`, `Hunk`, `DiffLine`, `ChangeKind`, `DiffStat`) and `parseDiff` parser, plus `Version` and `parseDottedVersion`, ported from the Rust `vcs-diff` crate.
- `VcsToolkit.Git`: the `git` CLI client (`Git`) ported from the Rust `vcs-git` crate — status (porcelain v1/v2), branches, commit/`commitPaths`, checkout, diff/log, merge/rebase/reset, fetch/push/clone, worktrees, tags, blame, config, remotes, cherry-pick/revert, `switchWithStash`, and a `harden` profile — with the builder specs (`CommitPaths`, `GitPush`, `MergeCommit`, `MergeNoCommit`, `WorktreeAdd`, `CloneSpec`, `AnnotatedTag`), validated `RefName`/`RevSpec` newtypes, `GitCapabilities`, and the git output parsers.
- `VcsToolkit.Jj`: the Jujutsu (`jj`) CLI client (`Jj`) ported from the Rust `vcs-jj` crate — changes/`log`, `describe`/`new`, bookmarks (local and remote-tracking), the operation log (`opLog`/`opRestore`/`opUndo`) with op-log-rollback `transaction`s, workspaces (incl. a bounded `workspaceRoots` fan-out), `squash`/`split`/`absorb`/`duplicate`/`abandon`, diff and template queries, and git sync (`gitFetch`/`gitPush`/`gitClone`/`gitImport`) — with the builder specs (`WorkspaceAdd`, `SquashPaths`), `JjFileset`/`RevsetExpr` newtypes, `JjCapabilities` gating on the validated jj >= 0.38 floor, and the jj output parsers.
- `VcsToolkit.GitHub`: the GitHub (`gh`) CLI client (`GitHub`) ported from the Rust `vcs-github` crate — pull requests (`prList`/`prView`/`prCreate`/`prMerge`/`prMarkReady`/`prClose`/`prReview`/`prComment`/`prEdit`/`prChecks`/`prFeedback`), issues (`issueList`/`issueCreate`/`issueView`), Actions runs (`runList`/`runView`/`runWatch`), releases (`releaseList`/`releaseView`), `repoView`, `authStatus`, and the raw REST/GraphQL `api` escape hatch — with the option types (`PrCreate`, `PrEdit`, `PrMerge`/`MergeStrategy`, `ReviewAction`/`ReviewKind`), the `--json` DTOs (`PullRequest`, `Issue`, `WorkflowRun`, `CheckRun`/`CheckBucket`, `Release`, `Review`/`Comment`/`PrFeedback`, `Repo`), and tolerant `System.Text.Json` parsers. Tokens resolve to `GH_TOKEN` (never argv).
- `VcsToolkit.GitLab`: the GitLab (`glab`) CLI client (`GitLab`) ported from the Rust `vcs-gitlab` crate — the lean merge-request lifecycle (`mrList`/`mrView`/`mrCreate`/`mrMerge`/`mrMarkReady`/`mrClose`/`mrComment`/`mrEdit`), CI/pipeline status (`mrChecks` → `CiStatus`), issues (`issueList`/`issueView`/`issueCreate`), releases (`releaseList`/`releaseView`), `repoView`, `authStatus`, and the REST/GraphQL `api` escape hatch — with the option types (`MrCreate`, `MrEdit`, `MergeStrategy`), the `--output json` DTOs (`MergeRequest`, `Repo`, `Issue`, `Release`, `CiStatus`), and tolerant parsers. `mrMerge` merges immediately (`--auto-merge=false`); tokens resolve to `GITLAB_TOKEN` (never argv).
- `VcsToolkit.Forge`: the unified forge facade ported from the Rust `vcs-forge` crate — one `Forge` handle (`GitHub`/`GitLab`/`Gitea`/`GitHubWithToken`/`GitLabWithToken`/`FromGitHub`/`FromGitLab`/`FromGitea`/`FromUnknown`) dispatches the shared PR/MR lifecycle (`AuthStatus`/`RepoView`/`PrList`/`PrView`/`PrCreate`/`PrComment`/`PrEdit`/`PrMerge`/`PrMarkReady`/`PrClose`/`PrChecks`, `Capabilities`, `IssueList`/`IssueView`/`IssueCreate`, `ReleaseList`/`ReleaseView`) to the underlying `VcsToolkit.GitHub` / `VcsToolkit.GitLab` / `VcsToolkit.Gitea` client — with the unified DTOs (`ForgeKind` + a security-hardened `OfRemoteUrl` host classifier, `ForgePr`/`ForgePrState`, `ForgeRepo`, `ForgeIssue`/`ForgeIssueState`, `ForgeRelease`, `CiStatus`, `PrCreate`, `MergeStrategy`, `PrEdit`, `ForgeCapabilities`, `ForgeOp`) normalising the three CLIs' shapes, and a `ForgeError` (`Forge`/`Unsupported`/`InvalidInput`) with `IsUnauthorized`/`IsRateLimited`/`IsTransientFetchError`/`IsTransient`/`IsNotFound`/`IsUnsupported` classifiers. A few ops (`RepoView`/`PrMarkReady`/`PrChecks`/`ReleaseView`/`PrDiff`) are `Unsupported` on Gitea; `Supports(ForgeOp)` branches before calling. The raw backend client is reachable via `GitHubClient`/`GitLabClient`/`GiteaClient` (`Some` only for the matching backend) for the per-forge operations off the shared surface — the `Repo.Git`/`Repo.Jj` escape-hatch analogue.
- `VcsToolkit.Core`: the backend-agnostic `Repo` facade ported from the Rust `vcs-core` crate — `Repo.Open` auto-detects git vs jj (`Detect.detect`: a pure `.git`/`.jj` upward probe, jj winning when colocated), then dispatches one common surface (`CurrentBranch`/`Trunk`/`LocalBranches`/`BranchExists`/`DeleteBranch`/`RenameBranch`, `ChangedFiles`/`DiffStat`/`HasUncommittedChanges`/`HasTrackedChanges`/`ConflictedFiles`/`Snapshot`, `CommitPaths`/`Fetch`/`FetchFrom`/`FetchBranch`/`Push`/`Checkout`/`Rebase`, `TryMerge`/`InProgressState`/`AbortInProgress`/`ContinueInProgress`, `ListWorktrees`/`CreateWorktree`/`RemoveWorktree`) to the underlying `VcsToolkit.Git` / `VcsToolkit.Jj` client — with the backend-agnostic DTOs (`BackendKind`, `FileChange`, `WorktreeInfo`, `OperationState`, `UpstreamTracking`, `RepoSnapshot`, `MergeProbe`, `CreateOutcome`) reusing `VcsToolkit.Diff`'s `ChangeKind`/`DiffStat`, and a `RepoError` (`NotARepository`/`WorktreeNotFound`/`Io`/`Vcs`) with `IsMergeConflict`/`IsNothingToCommit`/`IsLockContention`/`IsTransientFetchError`/`IsTransient`/`IsNotFound` classifiers. `Repo.FromGit`/`FromJj` are the test seam; `.Git`/`.Jj` are the raw-client escape hatches. The synchronous `cleanupWorktreeBlocking` Drop-guard helper is not ported (.NET's `IAsyncDisposable` can `await RemoveWorktree`).
- `VcsToolkit.Gitea`: the Gitea/Forgejo (`tea`) CLI client (`Gitea`) ported from the Rust `vcs-gitea` crate — the lean pull-request lifecycle (`prList`/`prView`/`prCreate`/`prMerge`/`prClose`/`prComment`/`prEdit`), issues (`issueList`/`issueView`/`issueCreate`), and release listing (`releaseList`) — with the option types (`PrCreate`, `PrEdit`, `MergeStrategy`), the DTOs (`PullRequest`, `Issue`, `Release`) parsed from tea's all-strings print-table (and the typed issue-detail object), and `authStatus` via the configured logins. Authentication is ambient (`tea`'s stored logins); `prView` is synthesized by listing and filtering.
- Forge CLI version parsing + `Capabilities` on the three forge wrappers: `GitHub.Capabilities()` / `GitLab.Capabilities()` / `Gitea.Capabilities()` probe `gh`/`glab`/`tea --version` and return a `GitHubCapabilities` / `GitLabCapabilities` / `GiteaCapabilities` carrying the parsed `VcsToolkit.Diff.Version`, an `IsSupported` floor check, `EnsureSupported()`, and the static `MinimumSupported` floor (gh ≥ 2.0, glab ≥ 1.0, tea ≥ 0.9). An unrecognisable `--version` banner degrades to a predictable `Parse` error, never a throw. (The three wrappers now also depend on `VcsToolkit.Diff` for the shared `Version` type.)
- `VcsToolkit.Forge`: `Forge.Capabilities()` now reports the detected CLI `Version` and backend `Kind` on `ForgeCapabilities` (surfaced independently of `Authed`, so the version shows even when unauthenticated; the `Unknown` handle reports `Version = None` / `Kind = Unknown`), and the mutating operations `PrCreate`/`PrMerge`/`PrEdit`/`IssueCreate` are version-gated — refused up front with the new structural `ForgeError.UnsupportedVersion` (carrying the forge, operation, found, and minimum versions) when the installed CLI is confirmed below the wrapper's floor, instead of failing on a raw CLI error. A version that can't be probed or parsed falls through (fail-open), and read-only operations are never gated.
- Packaging: every published package ships a NuGet package icon, and each package declares its inter-package `VcsToolkit.*` dependencies (so restoring one package pulls in the siblings it needs).
- `vcs-mcp`'s `repo_show_file` truncates content past a configurable output budget (`--output-budget <bytes>`, default 200000; `0` disables the cap) with an explicit `[truncated: showing N of M bytes]` marker, so a huge file read can no longer blow out an agent's context; content within the budget is returned byte-for-byte unchanged.
- `Git.EmptyTreeOid(dir)` (and `GitAt.EmptyTreeOid()`) — the empty-tree object id in a repository's actual object format, resolved via `git hash-object -t tree --stdin` fed empty input (nothing is written to the object database).
- Locally check out a pull/merge request's branch: `GitHub.PrCheckout` (`gh pr checkout`), `GitLab.MrCheckout` (`glab mr checkout`), `Gitea.PrCheckout` (`tea pr checkout`), the unified `Forge.PrCheckout` (dispatched across all three; `Unsupported` only on the CLI-less `Unknown` handle), and the write-gated `vcs-mcp` `forge_pr_checkout` tool. Unlike the remote-only forge operations this switches the local working tree to the PR/MR's source branch, so on the MCP server it serializes with the `repo_*` mutations.
- `VcsToolkit.Core`'s `OperationState` now models the remaining git sequencer states — `CherryPick` (`CHERRY_PICK_HEAD`), `Revert` (`REVERT_HEAD`), and `Bisect` (`BISECT_LOG`) — alongside `Merge`/`Rebase`/`ApplyMailbox`. `Repo.Snapshot`/`Repo.InProgressState` now detect them (keyed off each operation's own git-dir marker, so a cherry-pick/revert conflict is no longer mis-read as `Clear`), `Repo.AbortInProgress` routes `cherry-pick --abort` / `revert --abort` / `bisect reset`, and `Repo.ContinueInProgress` routes `cherry-pick --continue` / `revert --continue` (a re-conflict on the next commit surfaces as `Conflict`, not an error). A bisect has no continue step, so `Repo.ContinueInProgress` during a bisect is refused up front with the new `RepoError.Unsupported` (test with `IsUnsupported`) instead of silently doing nothing. The underlying `VcsToolkit.Git` client gains the matching probes (`IsCherryPickInProgress`/`IsRevertInProgress`/`IsBisectInProgress`) and commands (`CherryPickAbort`/`CherryPickContinue`/`RevertAbort`/`RevertContinue`/`BisectReset`), on both `Git` and the `GitAt` view. `VcsToolkit.Watch`'s `RepoEvent.OperationChanged` now reports transitions through these states too.
- `Jj.RollbackTo(dir, capturedOpHead)` (and `JjAt.RollbackTo(capturedOpHead)`) on `VcsToolkit.Jj` — the shared op-log rollback protocol behind `Jj.Transaction` and `Repo.TryMerge`, exposed for direct use. It runs its cleanup (`op log` + `op restore`) on a fresh cancellation token with its own timeout (never inheriting an already-cancelled operation's budget) and, before restoring, probes the recent op log for divergence: if a concurrent operation advanced the op-head past `capturedOpHead`, it refuses to restore rather than clobbering that work. The outcome is reported via the new `RollbackOutcome` type (`RolledBack` / `SkippedDiverged of captured * current`).
- `vcs-mcp`'s `--log-commands <stderr|path>` flag: a diagnostic log of every spawned git/jj/forge command (program, argv, cwd, retry attempt, duration, and outcome — exit code or classified error) written to stderr or the given file, for troubleshooting a hung, slow, or failing CLI call from inside an agent session where stdout is reserved for the MCP protocol. Off by default; secrets never appear in the log (only what `VcsToolkit.CliSupport`'s `ICommandObserver` contract exposes, which by construction carries no environment/credential values).
- `Jj.ReadOnly()` / `Jj.IsReadOnly` on `VcsToolkit.Jj` — a read-only view of the jj client (the analogue of jj's `WorkingCopy::Ignore`) whose read/query methods (`Status`/`Log`/`Diff…`/bookmark listings/`Root`/`op log` reads/workspace lookups/conflict probes) pass jj's global `--ignore-working-copy` flag, so a query reports the last recorded operation's state without snapshotting the working copy or advancing the operation log — jj otherwise records a new operation as a side effect of an ordinary read. Mutating methods are deliberately unaffected. The flag is a hard-coded literal placed before the subcommand, so it can never be injected from an untrusted revset/bookmark/operation id.
- `VcsToolkit.Forge`: `ForgePr` and `ForgeIssue` gain `Labels` and `Assignees` (`string list option`) — GitHub and GitLab report a confirmed `Some` (an empty `Some []` is a confirmed "no labels" / "unassigned"), while Gitea is always `None` (its `tea` CLI has no labels/assignees columns, so an empty list would be a false "none" rather than the honest "unknown"), mirroring the `None`/`Some []` contract of `ForgePr.Draft`. The underlying `VcsToolkit.GitHub` and `VcsToolkit.GitLab` `PullRequest`/`Issue`/`MergeRequest` DTOs also carry the flattened `Labels`/`Assignees` lists (`gh`'s `--json` field set now requests `labels,assignees`; `glab`'s `--output json` already returns them).
- Typed pull/merge request diffs: `GitHub.PrDiff(dir, number)` (`gh pr diff <n>`) and `GitLab.MrDiff(dir, number)` (`glab mr diff <n>`) return the PR/MR's unified diff parsed into per-file `VcsToolkit.Diff.FileDiff` values, and the unified `Forge.PrDiff(number)` dispatches to the matching backend — `Unsupported` on Gitea (`tea` has no diff command) and on the CLI-less `Unknown` handle.
- Path-scoped history ("who changed this module"): `Git.LogPaths(dir, revspec, max, paths)` and `Jj.LogPaths(dir, revset, max, filesets)` (plus the `GitAt`/`JjAt` views), the backend-agnostic `Repo.Log(revspecOrRevset, max)` / `Repo.LogPaths(revspecOrRevset, max, paths)` facade over the new unified `VcsToolkit.Core.Commit` DTO (`Author`/`Date` are `Some` on git, `None` on jj), and the read-only `vcs-mcp` `repo_log` tool. On git, `LogPaths` applies `--literal-pathspecs` (a `*`/`?`/`[]` in a path matches literally, on both the direct and the large-path-set path); since `git log` has no `--pathspec-from-file` transport, an over-budget path set is split across several `git log` calls whose results are merged (deduped by hash) and restored to git's native order via a pathless `--format=%H` oracle over the once-resolved revspec, and a single path that alone exceeds the argv budget or an empty path set is refused before spawning. On jj, paths become exact-path `root-file:"…"` filesets (workspace-root-relative, so they resolve the same whether the handle is bound to the repo root or a subdirectory) and an empty set is likewise refused.
- `Repo.OpenWith(dir, git, jj)` on `VcsToolkit.Core` — open a repository like `Repo.Open`, but build the client for the detected backend from an injected `unit -> Git` / `unit -> Jj` factory instead of the plain `Git.Create()`/`Jj.Create()` default, for callers that need a pre-configured client (e.g. a hardened, timeout-bound one) without re-implementing detection, path absolutisation, and error mapping. Only the factory for the backend actually detected is invoked, so configuring both costs nothing for the unused one; the path/detection/`RepoError` contract is identical to `Open` (`InvalidInput` on a bad path, `NotARepository` when no `.git`/`.jj` is found).
- `vcs-mcp` gains six write-gated `repo_*` tools covering the rest of the `Repo` facade: `repo_rebase` (rebase the current work onto a target), `repo_abort_in_progress` / `repo_continue_in_progress` (abort or continue an interrupted merge/rebase, reporting the fresh post-call operation state), `repo_delete_branch` / `repo_rename_branch` (local branch/bookmark management), and `repo_new_child` (start fresh work on top of a reference without rewriting it). Each is gated by the `WriteGate` (`--allow-write` / `--allow-tools`) and serializes on the per-repo write lock, with per-tool destructive/idempotent annotations.
- GitHub Enterprise Server (GHES) support on `VcsToolkit.GitHub`: a `GitHubHost` classifier (`GitHubHost.GitHubCom`, `GitHubHost.New(host)`, `GitHubHost.OfRemoteUrl(url)` with `Host`/`IsGitHubCom`/`IsEnterprise`) that treats `github.com` (any case) as SaaS and any other valid dotted host as GHES, canonicalising to a lower-cased hostname — an empty, malformed, or ambiguous host (an unparseable/hostless remote, an IPv6 literal, a bare single-label scp authority) is a diagnosable error, never a silent github.com fallback. `GitHub.WithHost(host)` (and the `GitHubAt` chain) binds a client to that host so a supplied credential is injected into the variable `gh` reads for it — `GH_TOKEN` for github.com, `GH_ENTERPRISE_TOKEN` for a GHES host — with `GH_HOST` pinned and the bound host carried in each operation's `CredentialRequest`, so an enterprise secret never lands in the github.com token env (nor vice versa). `GitHub.AuthStatusFor(host)` / `GitHubAt.AuthStatusFor(host)` scope the auth probe to one host (`gh auth status --hostname <host>`), so a broken session on a different host can't read as a false negative for the targeted one.
- The `vcs-mcp` Model Context Protocol server is now published as a .NET global tool — install it with `dotnet tool install --global vcs-mcp` and run it against a repository (`vcs-mcp --repo <path> [--allow-write]`).
- Diagnostic command observation across the whole toolkit: `VcsToolkit.CliSupport`'s `ManagedClient.WithObserver(observer)` attaches an opt-in `ICommandObserver` that is notified as each spawned CLI command starts and finishes — once per lock-contention retry attempt — carrying a `CommandEvent` (program, argv, working directory, 0-based attempt index, and a `HasSecret` flag) plus, on finish, the wall-clock duration and outcome (the exit code, or the `ProcessError` to classify). Secret values never reach the observer: only argv/cwd (which the wrappers keep tokens out of by construction) are exposed and `HasSecret` reports presence alone. It threads through every wrapper client via `Git.WithObserver` / `Jj.WithObserver` / `GitHub.WithObserver` / `GitLab.WithObserver` / `Gitea.WithObserver` (the same way `WithRetry`/`WithCredentials` do), giving one hook to log commands, measure durations, or diagnose slow/failing CLI calls without dropping into ProcessKit. With no observer attached the run path is unchanged.
- Complete the issue lifecycle across the forge surface: `GitHub.IssueClose`/`IssueComment` (`gh issue close` / `gh issue comment`), `GitLab.IssueClose`/`IssueComment` (`glab issue close` / `glab issue note`), `Gitea.IssueClose`/`IssueComment` (`tea issues close` / `tea comment`), the unified `Forge.IssueClose(number)` / `Forge.IssueComment(number, body)` (dispatched across all three; version-gated like `IssueCreate`, with an empty/whitespace comment body refused as `ForgeError.InvalidInput` before any spawn, and `Unsupported` only on the CLI-less `Unknown` handle), and the write-gated `vcs-mcp` `forge_issue_close` / `forge_issue_comment` tools. Both are remote-only mutations, so — unlike `forge_pr_checkout`/`forge_pr_merge`/`forge_pr_close` — they hold no per-repo write lock.

The underlying clients gain `GitLab.MrApprove`/`MrRevoke` (`glab mr approve`/`glab mr revoke`) and `Gitea.PrApprove`/`PrReject` (`tea pr approve`/`tea pr reject`), plus the write-gated `vcs-mcp` `forge_pr_review` tool (`kind` = approve/request_changes/comment, with `body` required for request_changes/comment). A remote-only mutation, so it holds no per-repo write lock.
- `RepoDiff.diff` on `VcsToolkit.Watch` — the pure snapshot-diff behind `RepoWatcher` is now public: pass a `(RepoSnapshot, branch list)` pair "before" and "after" (e.g. two `Repo.Snapshot` + `Repo.LocalBranches` re-queries) to get the same `RepoEvent list` `RepoWatcher` would emit, without standing up a `FileSystemWatcher`. Lets a polling-based monitor reuse the exact event semantics on environments where filesystem-watch signals are unreliable or unavailable (network disks, containers/NFS without inotify, WSL boundaries). `RepoWatcher`'s background loop now calls this same function as its only source of event semantics.
- `GitAt.ResolvedGitDir()` — the bound-view forwarder for `Git.ResolvedGitDir(dir)`, closing the last gap in `GitAt`'s completeness contract (every modelled `dir`-taking `Git` member now has a `GitAt` counterpart).
- `Forge.PrList`/`Forge.IssueList` gain a `PrListOptions`/`IssueListOptions`-taking overload (state filter + result cap) alongside the original zero-argument member, which is preserved unchanged (open, up to 100) for binary compatibility. `PrListOptions.State` (`Open`/`Closed`/`Merged`/`All`) maps to `gh pr list --state` and `glab mr list [--closed|--merged|--all]` directly. **Gitea is `Unsupported` for every state on both `PrList` and `IssueList`**, refused with `ForgeError.Unsupported` before any spawn — not just `Closed`/`Merged`: `tea`'s `pr list`/`issues list --output json` does not work against the real CLI at all, for any `--state` (a pre-existing gap, unrelated to this filter), so there is no working listing path to reach regardless of the requested state (see the `PrListState`/`IssueListState` doc comments). `IssueListOptions.State` is `Open`/`Closed`/`All` (issues have no merged state). The underlying `VcsToolkit.GitHub`/`VcsToolkit.GitLab`/`VcsToolkit.Gitea` clients gain the same `(dir)` / `(dir, options)` overload pair (`GitLab`'s options type is `MrListOptions`; the `VcsToolkit.Gitea` client's own `(dir, options)` overload still builds and sends the `tea` command as documented — the facade-level `Unsupported` refusal lives one layer up, in `VcsToolkit.Forge`). The `vcs-mcp` `forge_pr_list`/`forge_issue_list` tools gain a matching `(state, limit)` overload alongside the original zero-argument one (`limit` must be positive).

Dispatched to `gh pr list --head <branch> --state all` (a new two-argument `GitHub.PrListForBranch(dir, head)` overload alongside the existing three-argument `(dir, head, baseBranch)` form, added since the facade is never given a target branch) and the new `GitLab.MrListForBranch(dir, sourceBranch)` (`glab mr list --source-branch <branch> --all`); **`Unsupported` on Gitea** (`tea pr list --output json` does not work against the real CLI — same K-049 root cause as `PrList`). The branch name is checked against the argv injection guard before spawning on both GitHub and GitLab — including the pre-existing three-argument `GitHub.PrListForBranch`, which previously did not guard its `head`/`baseBranch` arguments. Also exposed as the read-only `vcs-mcp` `forge_pr_for_branch` tool (`source_branch`, required).

### Changed
- `VcsToolkit.Forge`: `Forge` gains the kind/variant-dependent capability queries `SupportsReview(kind)`, `SupportsMergeOptions`, and `SupportsCloseDeleteBranch` — the counterpart of `Supports(ForgeOp)` for the refusals `ForgeOp` can't express, so a consumer can now predict every `Unsupported` before calling: a `PrReview` kind unavailable on the backend (`RequestChanges` on GitLab; a `Comment`-review on GitLab/Gitea), `PrMerge`'s `Auto`/`DeleteBranch` options (GitHub only), and `PrClose`'s delete-branch (GitHub only). Introspection and dispatch now read one shared source, so they cannot drift. The `ForgeOp` doc comment is corrected accordingly (its former "every other operation is supported on all three forges" was stale).
- `Git.ResolvedGitDir(dir)` and `GitOperationMarkers.classify(gitDir)` expose the resolved git metadata directory and its interrupted-operation markers for consumers that need the same operation-state detection as the client.
- `Forge` constructors and `At` now absolutise `cwd` at bind time, matching `Repo`'s contract.
- MCP write-tool annotations now report each operation's destructive and idempotent semantics instead of using uniform defaults.
- `VcsToolkit.Forge`: `Forge.PrClose(..., deleteBranch = true)` now refuses GitLab and Gitea with `ForgeError.Unsupported` before spawning, instead of silently ignoring branch deletion; closing without branch deletion and GitHub behavior are unchanged.
- MCP clients now see `InvalidParams` instead of `Internal` for input validation errors from the core facade.
- `VcsToolkit.Jj`'s machine-parsed identity templates (`jj workspace list`, `jj bookmark list`/`-a`, and the reachable-bookmarks query) now frame every free-text field (a bookmark or workspace name) through jj's `.escape_json()`, so a name carrying a tab, newline, comma, or quote — e.g. a bookmark imported from git — round-trips exactly instead of splitting the row or shifting columns. Their identity commit ids (`Bookmark.Target`, `BookmarkRef.Target`, `Workspace.Commit`, and so the facade's `WorktreeInfo.Commit`) are now the **full** `commit_id` rather than a `.short()` prefix, so they compare directly against `RepoSnapshot.Head` / a git oid without a short-prefix collision. Consumers relying on the previous short `WorktreeInfo.Commit` should truncate for display. The history-display `jj log`/`evolog` rows keep their short ids by design.
- `VcsToolkit.CliSupport`/`VcsToolkit.Git` now scope credential resolution to the operation's known host so a multi-host `ICredentialProvider` serves the right secret. `ManagedClient` gains `WithExpectedHost(host)`, whose value becomes the `CredentialRequest.Host` on the token-env injection path (previously always `None`); and `git`'s remote operations pass the clone URL's host into the provider lookup (previously ignored — the host scoped only the credential helper, not the secret selection), so the one host drives both provider selection and the helper's host gate and they cannot disagree. The fallback policy is unchanged but now documented and covered for reads and writes: no provider / provider `Ok None` / an empty secret all defer to ambient auth, while a provider `Error` fails closed (the operation aborts instead of silently running unauthenticated).
- `VcsToolkit.Jj`'s `Jj.Transaction` rollback (and the jj path of `VcsToolkit.Core`'s `Repo.TryMerge`) is now hardened against cancellation and concurrency: the on-error / post-probe cleanup runs on a fresh cancellation budget, so a cancelled or timed-out closure is still rolled back instead of silently skipping restore, and a divergence probe refuses to `op restore` when a concurrent operation advanced the op-head past the captured point (for `TryMerge`, a diverged rollback surfaces as an error rather than a `MergeProbe` result that would misdescribe the tree). `Jj.Transaction` still returns the closure's original error unchanged.
- The git path of `VcsToolkit.Core`'s `Repo.TryMerge` (`GitBackend.tryMerge`'s three cleanup branches — success / merge-conflict / other-error) is now hardened against cancellation the same way as the jj path: the merge-in-progress probe and `merge --abort` cleanup run on a fresh, dedicated cancellation budget (the new `Git.IsMergeInProgressDetached` / `Git.MergeAbortDetached`, also exposed on `GitAt`) instead of inheriting the probe merge's own — possibly already-cancelled or timed-out — token. A cancelled or timed-out probe merge on either backend no longer risks leaving a staged merge in the working tree.
- `VcsToolkit.Forge`: `Forge.PrMerge` now takes a unified `PrMerge` merge-spec (`Strategy`/`Auto`/`DeleteBranch`, built via `PrMerge.Merge`/`Squash`/`Rebase` + `WithAuto`/`WithDeleteBranch`) instead of a bare `MergeStrategy`, so auto-merge and delete-source-branch are available off the forge-agnostic surface. On GitHub `Auto`/`DeleteBranch` map to gh's real `--auto`/`--delete-branch` flags; on GitLab and Gitea — whose CLIs expose no confirmed equivalent — a spec asking for either is refused with `ForgeError.Unsupported` before any spawn rather than silently dropping the option, while a plain strategy merge still works on all three. The `vcs-mcp` `forge_pr_merge` tool gains matching optional `auto`/`delete_branch` boolean parameters.
- `VcsToolkit.Forge`: the best-effort `ForgePr.Draft`, `ForgeRepo.Private`, `ForgeRelease.Draft`, and `ForgeRelease.Prerelease` fields change type from `bool` to `bool option`. `Some` now carries a value the backend actually reported and `None` means it does not surface the field, so an unreported draft/visibility is no longer collapsed into a confirmed `false`: `None` for a GitHub/Gitea PR draft, a GitLab release draft/pre-release, and an absent GitLab project visibility; `Some` elsewhere.
- `VcsToolkit.CliSupport`'s `Retry.retryAsync` now takes a `CancellationToken` (a new required parameter between `shouldRetry` and `op`) and observes it around the backoff sleep between attempts — a cancellation firing mid-backoff aborts the retry immediately (instead of sleeping out the remaining delay) and does not start another attempt, surfacing as `ProcessError.Cancelled`; `ManagedClient`'s retry paths now propagate their configured cancellation token through this backoff.
- `VcsToolkit.Watch`'s `RepoWatcher` background loop now retries a **transient** re-query failure (per `WatchError.IsTransient`) in place with a bounded, cancellable backoff instead of silently skipping it every debounce cycle, and treats a **terminal** failure (not transient, or the retry budget exhausted) as a hard stop: it closes the output channel with an error (`WatcherTerminated`, carrying the underlying `WatchError`) instead of skip-and-continue forever. `RepoWatcher.Recv()` now re-raises that error (as `ChannelClosedException.InnerException`) instead of returning `None`, so a consumer can tell "the watch broke" apart from an ordinary `Dispose` (which still yields `None`).
- `VcsToolkit.Mcp`'s `VcsMcpServer` constructor gains a fourth `outputBudget: int option` parameter (the output-size budget, in bytes, applied to `repo_show_file`; `None` means no limit) — a breaking change to the constructor's arity.
- The raw `Run`/`RunRaw` escape hatches on the directory-bound views (`GitAt`, `JjAt`, `GitHubAt`, `GitLabAt`, `GiteaAt`) now run in the view's bound `dir` — like the modelled methods — instead of the process's current directory, so `git.At(dir).Run [ … ]` invokes the CLI in `dir`. Each client (`Git`/`Jj`/`GitHub`/`GitLab`/`Gitea`) gains a `dir`-taking `Run(dir, args)`/`RunRaw(dir, args)` overload that backs this, while the no-`dir` `Run(args)`/`RunRaw(args)` on the unbound client still runs in the process cwd. This reverses the earlier "raw hatches deliberately stay in the process cwd" behaviour and is a breaking behavioural change for code that relied on a bound view's `Run`/`RunRaw` ignoring `dir`: call `Run`/`RunRaw` on the unbound client (or pass an explicit directory) for a process-cwd command.
- The pinned `ProcessKit`/`ProcessKit.Testing` dependency floor moves from 2.2.0 to 2.3.0. `VcsToolkit`'s own runtime and test-double usage is unaffected (no direct use of `RecordReplayOptions`/`RecordReplayRunner` cassettes, `WithCwdMatching()`, or `RunningProcess.WaitForAsync`/`WaitForPortAsync` anywhere in the source or tests — `ManagedClient`'s test seam is exercised only through `ScriptedRunner`/`Reply`). Consumers who inject their own `ProcessKit.Testing.RecordReplayRunner` (via `ManagedClient.WithRunner`/`IProcessRunner`) to record/replay cassettes against `VcsToolkit` clients should note that cassette matching by default no longer keys on the working directory (`Command.CurrentDir`) — pass `RecordReplayOptions.WithCwdMatching()` to restore the previous cwd-sensitive matching if two of their recorded entries relied on `cwd` alone to stay distinct.
- `vcs-mcp`'s `forge_pr_merge` tool now holds the same per-repo write lock as `forge_pr_checkout` and the `repo_*` mutations for its whole call (previously it only checked the write gate, without serializing): with `delete_branch` it can delete the local branch and switch the checkout, so it must not race a concurrent repo mutation on the same working copy.
- `vcs-mcp`'s `forge_pr_close` tool now holds the same per-repo write lock as `forge_pr_checkout`/`forge_pr_merge` and the `repo_*` mutations for its whole call (previously it only checked the write gate, without serializing): with `delete_branch` (and even without it, for symmetry with `forge_pr_merge`) it can touch the same working tree the `repo_*` mutations do, so it must not race a concurrent repo mutation.
- `VcsToolkit.Forge`'s version-gated mutating operations (`PrCreate`/`PrEdit`/`PrMerge`/`IssueCreate`) now probe the backend CLI's `--version` at most once per `Forge` handle — the result is cached for the handle's lifetime instead of being re-spawned on every gated call — and `Forge.Capabilities()` reuses the same cached probe rather than spawning its own. Fewer `gh`/`glab`/`tea --version` spawns per handle; behavior (fail-open on an unparsed version, `UnsupportedVersion` for a confirmed too-old CLI) is unchanged.
- `vcs-mcp` now maps its two `McpError` cases onto the two distinct MCP error channels, so a client can tell a "fix your call" refusal apart from a backend failure: an `InvalidParams` error (an unknown tool, a bad/missing argument, a disabled write tool, an unsupported forge op) is raised as a JSON-RPC **protocol** error carrying `McpErrorCode.InvalidParams`, while an `Internal` error (a git/jj/forge backend or network failure) is still returned inside the tool result with `IsError = true`. Previously both were flattened into an `IsError` result, losing the distinction the server's doc-comment already promised.

- forge auto-detection on jj repositories no longer fails silently when `ui.color = "always"` is set in the user's jj config.

- `VcsToolkit.Jj`'s internal `decodeJsonField` no longer emits a NUL character for a truncated `\uXXXX` escape (fewer than four hex digits); decoding now stops instead, matching the parser's own documented total-decoding contract.

- `Jj.CommitPaths` and `JjAt.CommitPaths` now refuse an empty fileset before spawning `jj commit`, preventing a direct client call from committing the entire working copy.
- `Git.Log`/`GitAt.Log` now preserve the full commit subject when it contains a literal 0x1f byte (e.g. a commit made with `--cleanup=verbatim`), instead of silently truncating it at the first occurrence.
- `GitHub.PrDiff` and `GitLab.MrDiff` now preserve the complete raw CLI diff output, including a trailing newline and a final blank context line, so their parsed `FileDiff.Raw` values remain byte-exact and the final hunk stays valid.
- `Repo.CommitPaths`/`Repo.LogPaths` now resolve their repo-relative input paths against the repository root on both backends even when the handle is bound to a subdirectory (`Repo.Open(<root>/sub)`), matching the root-relative paths `Repo.ChangedFiles` reports — so a path taken from `ChangedFiles` targets the exact same file. Previously the paths were resolved relative to the handle's working directory (git pathspecs from the bound `dir`; jj's cwd-relative `file:` fileset), so a subdirectory-bound handle could commit/log the wrong files or none. The git backend now runs `commit --only`/`log` from the repository root, and `VcsToolkit.Jj`'s `JjFileset.Path` now emits jj's workspace-root-relative `root-file:` anchor instead of the cwd-relative `file:` — which also anchors the other fileset consumers (`Jj.SquashPaths`/`Jj.SplitPaths`/`Jj.Absorb`/`Jj.FileShow`, and so `Repo.ShowFile`) to the workspace root.

- `Git.Add`/`Git.CommitPaths` now apply `--literal-pathspecs` unconditionally, so a path containing a glob metacharacter (`*`, `?`, `[]`) matches literally instead of being silently expanded as a pathspec pattern; a path set whose combined argv length would approach the OS command-line limit (Windows' ~32767-character `CreateProcess` ceiling) is now routed through the NUL-safe `--pathspec-from-file=- --pathspec-file-nul` stdin transport instead of risking a spawn failure, with a small path set still going through a single, argv-unchanged call.
- `GitLab`'s `MrCreate`/`MrEdit`/`IssueCreate`/`MrComment` now refuse a body/description of exactly `"-"` before spawning `glab`, instead of hanging on glab's own stdin/`$EDITOR` sentinel prompt.
- `Git.Upstream` no longer collapses a detached `HEAD` or a directory outside a repository into a false "no upstream configured" (`Ok None`); it now checks that `HEAD` is an attached branch first and surfaces those cases (and any `rev-parse @{u}` failure other than the documented "no upstream" exit code) as `Error`.
- `VcsToolkit.Jj`'s `JjFileset.Path` no longer unconditionally rewrites `\` to `/`: on non-Windows platforms a backslash is a legitimate filename byte, so it is now preserved and escaped as `\\` (instead of being silently turned into `/`, mismatching the real file, or left unbalanced at the end of the path); the `\`→`/` normalisation still applies only on Windows.
- `Jj.Status`/`Jj.DiffSummary` now resolve the workspace root and run the underlying `jj diff --summary` FROM it, instead of from whichever `dir` the caller bound — so the returned paths (including `OldPath` for a rename/copy) are workspace-root-relative regardless of which subdirectory `dir` names, matching the facade's repo-relative promise for the same DTO on the git backend, instead of returning cwd-relative paths (which could carry a literal `..` when called from a subdirectory). A reported path that still escapes the workspace root is now rejected as an error rather than silently propagated.
- `Git.DiffText`/`Diff` diffing an unborn (no-commits-yet) working tree no longer targets the hardcoded SHA-1 empty-tree constant, which does not exist as an object in a repository under `extensions.objectFormat=sha256`; it now resolves the empty-tree id via the new `Git.EmptyTreeOid`, which asks the repository's own `git hash-object` for it.
- `VcsToolkit.Watch`'s `RepoWatcher` now observes a jj repository read-only: its baseline query and every filesystem-driven re-query pass `--ignore-working-copy` (via the jj client's `ReadOnly()` view), so watching no longer snapshots the working copy or records a new jj operation on each debounce cycle. Previously every re-query mutated the very repository it was watching — taking the working-copy lock and churning the operation log the watch then reacted to. Git watching is unchanged.
- `VcsToolkit.Core`'s jj worktree cleanup is now diagnosable instead of silently masking failures. `Repo.RemoveWorktree`'s workspace-by-path lookup no longer collapses a *failed* `jj workspace root --name` probe into the same `RepoError.WorktreeNotFound` as a genuine miss — a probe failure now surfaces a distinct error naming the workspace(s) jj couldn't resolve. `Repo.CreateWorktree`'s rollback after a failed `BookmarkCreate` no longer swallows a subsequent directory-delete exception or a failed `WorkspaceForget`: either failure is now composed into the returned error alongside the original `BookmarkCreate` failure, instead of one masking the other. A pre-existing destination directory (not created by the failed call) is still never deleted by the rollback.
- `Git.Push`'s refspec guard now also refuses a refspec with an empty side (`":branch"`, `":"`, `"local:"`) before spawning, instead of silently letting it through to `git push` — `":branch"` deletes the remote branch and `":"` pushes all matching branches, both destructive fan-out/deletion the typed API's `branch`/`local:remote` contract claims impossible. A genuine delete/force-push must still go through the raw `Run [ "push"; … ]` escape hatch.
- `VcsToolkit.CliSupport`'s credential-helper host scoping now matches the URL scheme case-insensitively: a clone URL with an upper- or mixed-case scheme (`HTTPS://`, `Https://` — git/curl treat schemes case-insensitively) is now recognised as HTTPS, so its host is extracted and the git credential helper is scoped to it. Previously the scheme was compared byte-for-byte, so a mixed-case scheme on an externally-supplied clone URL silently produced an *unscoped* helper — leaking the secret to any host git subsequently contacted (a redirect or a submodule on a different host) during the clone. The host itself is still returned verbatim (case and port preserved); only the scheme comparison is case-folded.
- `vcs-mcp`'s `forge_pr_create` and `forge_issue_create` tools now refuse a `title`/`body` starting with `-` up front (as `InvalidParams`), matching the guard already applied to `forge_pr_comment`/`forge_pr_edit`, instead of only relying on the underlying CLI wrapper's own flag-injection defence.
- `vcs-mcp`'s forge auto-detection now resolves the `origin` remote on a **non-colocated** jj repository (`jj git init` without colocation) too: previously it only asked git's `remote get-url` at the repository root, which finds nothing when there's no root `.git`, so every jj user without colocation silently got "no forge is configured" regardless of a configured remote. It now falls back to `jj git remote list` on jj-backed repos. Git-backed repos and the explicit `--forge` override are unaffected.
- Closed an argv-injection gap in the git/jj client wrappers: several caller-supplied values that land in a bare positional slot were passed through unguarded, so a `-`-leading value could be parsed as a flag. `Git.CloneRepo` now guards `dest` (in addition to `url`) — a `dest` like `--upload-pack=<cmd>` was a command-execution vector, since git accepts options after positionals — `Git.WorktreeAdd` guards `spec.Path`, `Git.WorktreeRemove` guards `path`, `Git.WorktreeMove` guards `fromPath`/`toPath`, and `Jj.GitClone` guards `dest` and `Jj.WorkspaceAdd` guards `spec.Path`; each refuses an empty/`-`-leading value before spawning. `Git.ConfigSet` — whose `value` may legitimately begin with `-` (e.g. `-1`) — now emits an end-of-options `--` separator (`git config -- <key> <value>`) so the value is taken verbatim instead of being parsed as a flag (an attacker-supplied `--global`/`--unset` value can no longer redirect or subvert the write); its `key` stays guarded. Valid calls' argv is otherwise unchanged.
- `VcsToolkit.Git`'s `RefName.Create` now enforces the full load-bearing core of `git check-ref-format` it documented, closing a ref-confusion gap on untrusted branch/tag/remote names. It now also rejects a slash-separated component that begins with `.` (`feature/.hidden`) or ends with `.lock` (`foo.lock/bar`) — not only when the whole name does — a name ending with `.`, the reflog/upstream sequence `@{` (e.g. `main@{u}`, which git resolves below the ref layer and could redirect an operation to a different ref than the one validated), the single character `@`, and an empty path component (a leading or trailing `/`, or `//`). Valid names such as `main`, `feature/x`, `a.b`, and `v1.2.3` are unaffected.
- `VcsToolkit.Watch`'s colocation probe (`Paths.stateDirs`) now recognizes a `.git` next to `.jj` as colocation only via the same valid-marker rule `VcsToolkit.Core.Detect` uses (a `.git` directory, or a gitlink file whose content starts with `gitdir:`), instead of treating any file or directory merely named `.git` as colocation. Previously a stray, non-gitlink `.git` file next to a valid `.jj` repository made `RepoWatcher.Build` fail with `WatchError.Notify` (`FileSystemWatcher`'s constructor rejects a file path), even though the repository itself was a perfectly valid jj repo. `Detect.isGitMarker` is no longer `private`, so `Watch` can call it directly.
- `VcsToolkit.Gitea`'s `AuthStatus`/`PrList`/`PrView`/`IssueList`/`IssueView`/`ReleaseList` now drive `tea`'s supported `--output csv` format instead of `--output json`, which `tea` 0.9.2 does not support on these commands (it returns `unknown output type 'json'` at exit 0, which the previous JSON parser then rejected) — so these operations work against the real `tea` CLI rather than always failing to parse. The quoted `outputdsv` tables are parsed positionally, empty and header-only listings read as an empty list, and an unsupported-output diagnostic surfaces as an error rather than a silent empty result. `IssueView` is now synthesized by paging `tea issues list` (tea 0.9.2's bare-index issue view renders Markdown and ignores `--output`). Public signatures are unchanged; the `Forge`-facade Gitea `PrList`/`IssueList` continue to report `Unsupported`.
