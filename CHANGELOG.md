# Changelog

and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added
- Hermetic conflict-marker models (pure, no subprocess) for reading a conflicted file's content into typed regions and writing a chosen resolution back. `VcsToolkit.Git.Conflict` handles git's `merge`/`diff3`/`zdiff3` styles (`ResolutionSide`, `ConflictRegion`, `ConflictSegment`, `hasConflictMarkers`/`parseConflicts`/`render`/`resolve`), and `VcsToolkit.Jj.Conflict` handles jj's native `diff`/`snapshot` materialized styles (`JjConflictSection`, `JjConflictRegion` with `Sides`/`Base`, `JjConflictSegment`, `JjResolution`, and the same function set) — a jj file materialized with jj's `git` style is redirected to the git model. Both `render` is a byte-exact roundtrip of `parseConflicts` (verbatim endings, CRLF, and a missing trailing newline preserved), and `resolve` honors jj's missing-terminating-newline encoding so a written-back resolution is byte-exact. Drive them from the already-exposed conflicted-path listers (`Repo.ConflictedFiles`, `Git.ConflictedFiles`, `Jj.ResolveList`) plus `Git.ShowFile` / `Jj.FileShow`.
- Directory-bound `at(dir)` views on every client and the `Repo` facade — an ergonomic layer over the `dir`-taking methods for code that works in one directory. `Git.At(dir)` / `Jj.At(dir)` / `GitHub.At(dir)` / `GitLab.At(dir)` / `Gitea.At(dir)` return a `GitAt` / `JjAt` / `GitHubAt` / `GitLabAt` / `GiteaAt` whose modelled methods drop the leading `dir` argument (so `git.At(dir).Status()` is exactly `git.Status dir`, byte-identical argv), while the raw `Run`/`RunRaw` escape hatches deliberately stay bound to the process's current directory (not `dir`). `JjAt.Transaction` forwards the op-log-rollback scope with `dir` pre-bound. `Repo.GitAt` / `Repo.JjAt` hand out the underlying client view bound to the handle's `Cwd` (`Some` only for the matching backend); re-anchor with `repo.At(path)` first to bind another worktree.
- `VcsToolkit.Mcp` + the `vcs-mcp` binary (`VcsToolkit.Mcp.Server`): a Model Context Protocol server exposing the toolkit's typed git/jj + forge operations as agent-callable tools, ported from the Rust `vcs-mcp` crate. The `VcsToolkit.Mcp` library holds the hermetically-testable core: `VcsMcpServer` with 32 tools — the `repo_*` group (`repo_snapshot`/`repo_info`/`repo_status`/`repo_diff_stat`/`repo_branches`/`repo_current_branch`/`repo_conflicts`/`repo_worktrees` reads; `repo_try_merge`/`repo_commit`/`repo_checkout`/`repo_fetch`/`repo_push`/`repo_create_worktree`/`repo_remove_worktree` writes) over `VcsToolkit.Core.Repo`, and the `forge_*` group (`forge_auth_status`/`forge_repo_view`/`forge_info`/`forge_pr_list`/`forge_pr_view`/`forge_pr_checks`/`forge_issue_list`/`forge_issue_view`/`forge_release_list`/`forge_release_view` reads; `forge_issue_create`/`forge_pr_create`/`forge_pr_merge`/`forge_pr_close`/`forge_pr_mark_ready`/`forge_pr_comment`/`forge_pr_edit` writes) over `VcsToolkit.Forge.Forge`. Read tools are always available; the 14 mutating tools are gated by a `WriteGate` (`None`/`All`/`Set`) driven by `--allow-write`/`--allow-tools`, and repo mutations serialize on a per-server lock. Also the tool catalogue with generated JSON-Schema input schemas + the `Catalog.callTool` dispatcher, an `McpError` with `coreErr`/`forgeErr`/`guardArgvField` mapping onto MCP's invalid-params/internal-error codes, and the `Args` CLI parser (`--repo`/`--forge`/`--allow-write`/`--allow-tools`/`--timeout`/`--help`). The `vcs-mcp` binary serves over stdio via the `ModelContextProtocol` SDK, auto-detects the forge from the `origin` remote (or `--forge`), and uses a hardened, timeout-bound git client. (JSON output is camelCase; the `VcsToolkit.Mcp` layer adds `FSharp.SystemTextJson`, and the binary adds `ModelContextProtocol` + `Microsoft.Extensions.Hosting`.)
- `VcsToolkit.Watch`: filesystem-watch a git/jj repository and emit typed state-change events, ported from the Rust `vcs-watch` crate — a `RepoWatcher` (`Watch`/`Builder` with `WorkingTree`/`Debounce`/`MaxWait`/`RequeryTimeout`) watches the `.git`/`.jj` state dir (resolving worktree gitlinks + the shared `commondir`), debounces the write burst (with a `MaxWait` ceiling), re-queries `Repo.Snapshot` + `LocalBranches` (bounded by `RequeryTimeout`), and diffs against the previous state to yield `RepoChange` (fresh snapshot + non-empty `RepoEvent` list). `RepoEvent` covers `HeadMoved`/`BranchSwitched`/`BranchCreated`/`BranchDeleted`/`WorkingCopyChanged`/`UpstreamChanged`/`AheadBehindChanged`/`OperationChanged`/`ConflictChanged`. Pull changes with `Recv`, read the last snapshot via `Current`, and monitor health via `Stats` (`WatcherStats`: requeries/changes/skips/`LastError`/watch-errors). A `WatchError` (`Notify`/`Vcs`/`Io`) carries `IsTransient`/`IsNotFound`/`ProcessError`. Built on `System.IO.FileSystemWatcher` + `System.Threading.Channels` (the .NET analogues of the Rust `notify` + tokio stack); `Dispose` stops the watch and the background loop.
- `VcsToolkit.TestKit`: throwaway git/jj sandboxes for integration tests, ported from the Rust `vcs-testkit` crate — a self-cleaning `TempDir` (unique via pid + a process-wide counter, removed on `Dispose`), `GitSandbox` (`Init`/`Write`/`AddAll`/`Commit`/`CommitFile`/`Branch`/`Checkout`/`RevParse`), `JjSandbox` (`Init`/`Write`/`Describe`/`NewChange`/`Bookmark`), a seeded `BareRemote` (a local clone/fetch/push source), and the raw `Raw.git`/`Raw.jj`/`Raw.configureIdentity` steps for directories no sandbox owns. Dependency-free (references no other `VcsToolkit` package, so it can be a test dependency of any of them without a cycle), hermetic (every git/jj invocation is isolated from the host's VCS config), and raising on failure. Runs the actual `git`/`jj` on `PATH`.
- `VcsToolkit.CliSupport`: shared plumbing ported from the Rust `vcs-cli-support` crate — the `rejectFlagLike` argv injection guard, error classifiers (`isMergeConflict`, `isNothingToCommit`, `isTransientFetchError`, `isLockContention`), the `RetryPolicy` / `Retry.retryAsync` lock-contention retry, credential provisioning (`Secret`, `Credential`, `ICredentialProvider`, `StaticCredential`, `EnvToken`, `Credentials.gitCredentialHelper`), the `ManagedClient` wrapper over the ProcessKit runner, and the tolerant, total `Json` helpers (`Json.strOr`/`strOpt`/`strFirst`/`u64Or`/`boolOr`/`nestedStr`/`arrayOf`/`withDoc`/`parseArray`/`parseArrayResult`/`parseObject`) that the forge wrappers build their `--json` parsers on.
- `VcsToolkit.Diff`: the git-format unified-diff model (`FileDiff`, `Hunk`, `DiffLine`, `ChangeKind`, `DiffStat`) and `parseDiff` parser, plus `Version` and `parseDottedVersion`, ported from the Rust `vcs-diff` crate.
- `VcsToolkit.Git`: the `git` CLI client (`Git`) ported from the Rust `vcs-git` crate — status (porcelain v1/v2), branches, commit/`commitPaths`, checkout, diff/log, merge/rebase/reset, fetch/push/clone, worktrees, tags, blame, config, remotes, cherry-pick/revert, `switchWithStash`, and a `harden` profile — with the builder specs (`CommitPaths`, `GitPush`, `MergeCommit`, `MergeNoCommit`, `WorktreeAdd`, `CloneSpec`, `AnnotatedTag`), validated `RefName`/`RevSpec` newtypes, `GitCapabilities`, and the git output parsers.
- `VcsToolkit.Jj`: the Jujutsu (`jj`) CLI client (`Jj`) ported from the Rust `vcs-jj` crate — changes/`log`, `describe`/`new`, bookmarks (local and remote-tracking), the operation log (`opLog`/`opRestore`/`opUndo`) with op-log-rollback `transaction`s, workspaces (incl. a bounded `workspaceRoots` fan-out), `squash`/`split`/`absorb`/`duplicate`/`abandon`, diff and template queries, and git sync (`gitFetch`/`gitPush`/`gitClone`/`gitImport`) — with the builder specs (`WorkspaceAdd`, `SquashPaths`), `JjFileset`/`RevsetExpr` newtypes, `JjCapabilities` gating on the validated jj >= 0.38 floor, and the jj output parsers.
- `VcsToolkit.GitHub`: the GitHub (`gh`) CLI client (`GitHub`) ported from the Rust `vcs-github` crate — pull requests (`prList`/`prView`/`prCreate`/`prMerge`/`prMarkReady`/`prClose`/`prReview`/`prComment`/`prEdit`/`prChecks`/`prFeedback`), issues (`issueList`/`issueCreate`/`issueView`), Actions runs (`runList`/`runView`/`runWatch`), releases (`releaseList`/`releaseView`), `repoView`, `authStatus`, and the raw REST/GraphQL `api` escape hatch — with the option types (`PrCreate`, `PrEdit`, `PrMerge`/`MergeStrategy`, `ReviewAction`/`ReviewKind`), the `--json` DTOs (`PullRequest`, `Issue`, `WorkflowRun`, `CheckRun`/`CheckBucket`, `Release`, `Review`/`Comment`/`PrFeedback`, `Repo`), and tolerant `System.Text.Json` parsers. Tokens resolve to `GH_TOKEN` (never argv).
- `VcsToolkit.GitLab`: the GitLab (`glab`) CLI client (`GitLab`) ported from the Rust `vcs-gitlab` crate — the lean merge-request lifecycle (`mrList`/`mrView`/`mrCreate`/`mrMerge`/`mrMarkReady`/`mrClose`/`mrComment`/`mrEdit`), CI/pipeline status (`mrChecks` → `CiStatus`), issues (`issueList`/`issueView`/`issueCreate`), releases (`releaseList`/`releaseView`), `repoView`, `authStatus`, and the REST/GraphQL `api` escape hatch — with the option types (`MrCreate`, `MrEdit`, `MergeStrategy`), the `--output json` DTOs (`MergeRequest`, `Repo`, `Issue`, `Release`, `CiStatus`), and tolerant parsers. `mrMerge` merges immediately (`--auto-merge=false`); tokens resolve to `GITLAB_TOKEN` (never argv).
- `VcsToolkit.Forge`: the unified forge facade ported from the Rust `vcs-forge` crate — one `Forge` handle (`GitHub`/`GitLab`/`Gitea`/`GitHubWithToken`/`GitLabWithToken`/`FromGitHub`/`FromGitLab`/`FromGitea`/`FromUnknown`) dispatches the shared PR/MR lifecycle (`AuthStatus`/`RepoView`/`PrList`/`PrView`/`PrCreate`/`PrComment`/`PrEdit`/`PrMerge`/`PrMarkReady`/`PrClose`/`PrChecks`, `Capabilities`, `IssueList`/`IssueView`/`IssueCreate`, `ReleaseList`/`ReleaseView`) to the underlying `VcsToolkit.GitHub` / `VcsToolkit.GitLab` / `VcsToolkit.Gitea` client — with the unified DTOs (`ForgeKind` + a security-hardened `OfRemoteUrl` host classifier, `ForgePr`/`ForgePrState`, `ForgeRepo`, `ForgeIssue`/`ForgeIssueState`, `ForgeRelease`, `CiStatus`, `PrCreate`, `MergeStrategy`, `PrEdit`, `ForgeCapabilities`, `ForgeOp`) normalising the three CLIs' shapes, and a `ForgeError` (`Forge`/`Unsupported`/`InvalidInput`) with `IsUnauthorized`/`IsRateLimited`/`IsTransientFetchError`/`IsTransient`/`IsNotFound`/`IsUnsupported` classifiers. A few ops (`RepoView`/`PrMarkReady`/`PrChecks`/`ReleaseView`) are `Unsupported` on Gitea; `Supports(ForgeOp)` branches before calling.
- `VcsToolkit.Core`: the backend-agnostic `Repo` facade ported from the Rust `vcs-core` crate — `Repo.Open` auto-detects git vs jj (`Detect.detect`: a pure `.git`/`.jj` upward probe, jj winning when colocated), then dispatches one common surface (`CurrentBranch`/`Trunk`/`LocalBranches`/`BranchExists`/`DeleteBranch`/`RenameBranch`, `ChangedFiles`/`DiffStat`/`HasUncommittedChanges`/`HasTrackedChanges`/`ConflictedFiles`/`Snapshot`, `CommitPaths`/`Fetch`/`FetchFrom`/`FetchBranch`/`Push`/`Checkout`/`Rebase`, `TryMerge`/`InProgressState`/`AbortInProgress`/`ContinueInProgress`, `ListWorktrees`/`CreateWorktree`/`RemoveWorktree`) to the underlying `VcsToolkit.Git` / `VcsToolkit.Jj` client — with the backend-agnostic DTOs (`BackendKind`, `FileChange`, `WorktreeInfo`, `OperationState`, `UpstreamTracking`, `RepoSnapshot`, `MergeProbe`, `CreateOutcome`) reusing `VcsToolkit.Diff`'s `ChangeKind`/`DiffStat`, and a `RepoError` (`NotARepository`/`WorktreeNotFound`/`Io`/`Vcs`) with `IsMergeConflict`/`IsNothingToCommit`/`IsLockContention`/`IsTransientFetchError`/`IsTransient`/`IsNotFound` classifiers. `Repo.FromGit`/`FromJj` are the test seam; `.Git`/`.Jj` are the raw-client escape hatches. The synchronous `cleanupWorktreeBlocking` Drop-guard helper is not ported (.NET's `IAsyncDisposable` can `await RemoveWorktree`).
- `VcsToolkit.Gitea`: the Gitea/Forgejo (`tea`) CLI client (`Gitea`) ported from the Rust `vcs-gitea` crate — the lean pull-request lifecycle (`prList`/`prView`/`prCreate`/`prMerge`/`prClose`/`prComment`/`prEdit`), issues (`issueList`/`issueView`/`issueCreate`), and release listing (`releaseList`) — with the option types (`PrCreate`, `PrEdit`, `MergeStrategy`), the DTOs (`PullRequest`, `Issue`, `Release`) parsed from tea's all-strings print-table (and the typed issue-detail object), and `authStatus` via the configured logins. Authentication is ambient (`tea`'s stored logins); `prView` is synthesized by listing and filtering.
- Packaging: every published package ships a NuGet package icon, and each package declares its inter-package `VcsToolkit.*` dependencies (so restoring one package pulls in the siblings it needs).
